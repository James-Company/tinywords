# 17 UI STYLE NOTEBOOK THEME

이 문서는 TinyWords UI 스타일 SSOT이다.  
핵심 톤은 **"과장되지 않은, 심플한 수첩(notebook) 감성"**이며, 학습 행동(시작/회상/복습)을 빠르게 유도하는 실용 중심 디자인을 목표로 한다.

---

## 1) 디자인 원칙

- **Calm:** 눈에 거슬리는 강한 대비/과도한 애니메이션을 지양한다.
- **Clear:** 지금 해야 할 행동(CTA)과 상태를 즉시 이해할 수 있어야 한다.
- **Consistent:** Today/Inbox/History/Settings 간 컴포넌트 규칙을 통일한다.
- **Encouraging:** 카피와 시각 요소는 압박보다 복귀와 지속을 돕는다.
- **Accessible:** 색상만으로 상태를 전달하지 않는다.

---

## 2) 브랜드 무드 키워드

- 조용함
- 따뜻함
- 정돈됨
- 손글씨 노트의 친근함
- 실용적 미니멀

금지 키워드:
- 네온/사이버
- 과도한 3D 글로우
- 게이미피케이션 과몰입 스타일

---

## 3) 컬러 시스템(Design Tokens)

### 3.1 기본 팔레트 (Light)

- `bg.primary`: `#F7F4ED` (종이톤 배경)
- `bg.surface`: `#FFFDF8` (카드 배경)
- `text.primary`: `#2A241E`
- `text.secondary`: `#6B6258`
- `line.default`: `#E5DDD1`
- `brand.primary`: `#6E5B45` (노트 브라운)
- `brand.soft`: `#EDE3D5`

### 3.2 상태 색

- `state.success`: `#3C7A57`
- `state.warning`: `#C08A2E`
- `state.error`: `#B85042`
- `state.info`: `#4A6FA5`

규칙:
- 상태색은 배경 틴트 + 아이콘/텍스트와 함께 사용
- `error`는 경고 목적, 사용자 비난 톤 금지

### 3.3 다크 모드 (선택적 MVP)

- 다크 모드가 없으면 Light 고정
- 지원 시 종이톤 컨셉 유지(완전 블랙 지양)

---

## 4) 타이포그래피

### 4.1 폰트

- 기본: 플랫폼 시스템 폰트
- 숫자/지표: 가독성이 높은 산세리프 유지

### 4.2 타입 스케일

- `display`: 28 / semibold
- `h1`: 24 / semibold
- `h2`: 20 / semibold
- `title`: 18 / medium
- `body`: 16 / regular
- `caption`: 13 / regular

규칙:
- 본문 최소 15~16px 유지
- 상태 설명/보조 문구는 `caption` 이상

---

## 5) 레이아웃/간격 시스템

- 기본 spacing scale: `4, 8, 12, 16, 20, 24, 32`
- 화면 좌우 기본 패딩: `16`
- 섹션 간 간격: `24`
- 카드 내부 패딩: `16`
- 리스트 아이템 간 간격: `12`

터치 타깃:
- 최소 `44x44pt`

---

## 6) 형태(Shape)와 그림자

- 기본 radius: `12`
- 강조 카드 radius: `16`
- pill/chip radius: `999`

그림자:
- 기본 카드: 매우 약한 그림자 1단
- 모달/바텀시트: 2단 그림자 허용
- 과도한 블러/글로우 금지

테두리:
- 카드/입력 기본 1px `line.default`

---

## 7) 컴포넌트 스타일 규칙

### 7.1 버튼

- Primary:
  - 배경 `brand.primary`, 텍스트 흰색
- Secondary:
  - 배경 `brand.soft`, 텍스트 `text.primary`
- Tertiary:
  - 텍스트 버튼, 밑줄 없음

상태:
- `disabled`: 명도 낮춤 + 클릭 차단
- `loading`: 라벨 유지 + 스피너

### 7.2 카드

- 구조: 제목, 보조정보, 주요 행동 CTA
- 텍스트 줄 수 제한:
  - 제목 최대 2줄
  - 보조 설명 최대 2줄

### 7.3 상태 칩(Chip)

- `pending`, `done`, `fail`, `overdue` 등은 아이콘 + 텍스트 동시 표기
- 색상만으로 의미 전달 금지

### 7.4 입력/폼

- 포커스 시 외곽선 명확히 표시
- 에러 시:
  - 빨간색만 사용하지 말고 안내 문구 포함

---

## 8) 화면별 톤 적용

### Today

- 가장 선명한 CTA(시작/이어하기)
- 진행률 시각화는 단순 바 형태 우선

### Inbox

- overdue는 눈에 띄되 위협적이지 않게
- "정리" 느낌의 안정적 리스트 리듬 유지

### History

- 회고 중심, 평가 느낌 최소화
- 연속성/공백을 사실적으로 보여주되 복귀 유도 문구 제공

### Settings

- 실용적 정보 구조 우선
- 위험 액션(초기화) 시각적 구분 명확히

---

## 9) 아이콘/일러스트 규칙

- 아이콘 스타일: 단순 라인 아이콘
- 이모지 남용 금지
- 일러스트는 보조 요소로만 사용(핵심 정보 대체 금지)
- 캐릭터/화려한 애니메이션은 MVP 범위 밖

---

## 10) 모션 가이드

- 기본 전환: 150~220ms
- easing: 부드러운 ease-out
- 반복/장식 애니메이션 최소화
- 로딩 애니메이션은 피드백 목적에 한정

금지:
- 화면 진입 시 과도한 스케일/회전
- 완료 시 폭죽형 과장 애니메이션

---

## 11) 카피 톤 가이드

- 톤: 조용한 코치, 비난 없는 문장
- 원칙:
  - 짧고 행동 중심
  - 실패 시 복귀 경로 제시
  - 압박/죄책감 유발 문구 금지

좋은 예:
- "오늘은 3개만, 확실하게."
- "괜찮아요. 오래된 복습부터 차근히 정리해요."

피해야 할 예:
- "연속이 깨졌어요. 지금 안 하면 늦어요."

---

## 12) 접근성 기준

- 텍스트 대비: WCAG AA 수준 권장
- 상태 전달: 색 + 텍스트 + 아이콘 병행
- 스크린리더 라벨:
  - 버튼/필터/상태 칩 모두 제공
- 동적 콘텐츠:
  - 로딩/오류/완료 메시지 접근성 알림 처리

---

## 13) 디자인 QA 체크리스트

- [ ] CTA가 한 화면에 1순위로 명확한가
- [ ] 상태가 색상만으로 전달되지 않는가
- [ ] spacing/radius/token이 규칙을 따르는가
- [ ] 비난형 카피가 포함되지 않았는가
- [ ] 작은 화면에서도 가독성이 유지되는가
- [ ] 터치 타깃 최소 크기를 만족하는가

---

## 14) 토큰 네이밍 규칙

- 색상: `color.{category}.{role}`
  - 예: `color.bg.primary`, `color.state.success`
- 타이포: `type.{role}.{size}`
  - 예: `type.body.16`
- 간격: `space.{n}`
  - 예: `space.16`
- 반경: `radius.{n}`
  - 예: `radius.12`

---

## 15) 확장 규칙(미래 테마)

- 시즌 테마 추가 시:
  - 정보 구조/컴포넌트 규칙은 유지
  - 색상/일러스트만 제한적으로 변경
- 테마 실험 A/B 시:
  - 학습 완료율/복귀율 저하가 있으면 롤백

---

## 16) 수용 기준(DoD)

1. 핵심 4개 화면(Today/Inbox/History/Settings)에 동일 토큰 체계가 적용된다.
2. 상태 표현은 색상 + 텍스트(필요시 아이콘) 조합을 만족한다.
3. 버튼/카드/칩 스타일 규칙이 화면 간 일관되다.
4. 카피 톤이 "격려형/행동형" 원칙을 준수한다.
5. 과한 애니메이션/과장 그래픽 없이 수첩 톤이 유지된다.
