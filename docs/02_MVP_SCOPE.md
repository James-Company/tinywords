# 02 MVP SCOPE

이 문서는 TinyWords MVP에서 **반드시 제공할 것(Must)**, **가능하면 제공할 것(Should)**, **MVP에서 제외할 것(Won't)** 을 정의한다.  
판단 기준은 "하루 3-5개 단어를 회상 중심으로 꾸준히 학습하게 하는가"이다.

---

## 1) MVP 목표

MVP의 목표는 기능을 많이 넣는 것이 아니라, 아래 루프가 매일 안정적으로 반복되게 하는 것이다.

1. 오늘 학습량 확인(3-5개)
2. 단어 학습(의미/회상/문장/발화)
3. 복습 큐 자동 생성 및 수행
4. 학습 기록 저장 및 스트릭 반영

즉, TinyWords의 MVP는 "작은 학습 루틴의 완결성"을 검증하는 단계다.

---

## 2) 범위 원칙

- **루틴 우선:** 학습의 시작-완료-복습 흐름이 끊기지 않아야 한다.
- **회상 우선:** 읽기/보기보다 떠올리기 행동을 우선 지원해야 한다.
- **마찰 최소화:** 하루 학습 시작까지 탭 수와 인지부하를 줄인다.
- **오프라인 우선:** 네트워크 없이도 핵심 학습 흐름이 가능해야 한다.
- **정책 일관성:** 복습/스트릭 규칙은 SSOT 문서와 일치해야 한다.

---

## 3) Must Have (MVP 필수)

### 3.1 학습 코어

- 하루 학습량 설정: 기본 3개, 사용자 설정으로 3-5개
- Today 화면에서 오늘 학습 대상과 진행 상태 표시
- 단어 카드 학습: 단어, 뜻, 품사, 예문(최소 1개)
- 회상 단계: 뜻/단어를 보고 반대편을 떠올리는 상호작용
- 발화 단계: 음성 녹음(최소 동작) 및 재생

### 3.2 복습/기록

- 복습 큐 생성: 학습 완료 후 정책에 따라 복습 일정 등록
- Inbox 화면에서 오늘 복습 대상 확인 및 수행
- History 화면에서 학습/복습 수행 이력 확인
- 스트릭 계산: 당일 완료 조건 충족 시 연속 일수 반영

### 3.3 생성/시스템

- AI 단어 생성 프롬프트 기반으로 일일 단어 후보 생성
- AI 문장 코치 프롬프트 기반의 예문/코칭 텍스트 생성
- 로컬 저장소 기반 데이터 유지(앱 재실행 후 복원)
- 핵심 화면 4종 제공: Today / Inbox / History / Settings

### 3.4 기본 안정성

- 오프라인 상태에서 기존 데이터 조회 및 학습 진행 가능
- 예외 처리: AI 실패/네트워크 오류 시 재시도 또는 대체 안내
- 최소 테스트 체계: 핵심 사용자 플로우 E2E 또는 통합 테스트

---

## 4) Should Have (가능하면 포함)

- 발음 점수의 단순 버전(정밀도보다 피드백 존재 여부 우선)
- 학습 완료 후 짧은 요약 피드백(오늘 달성 내용)
- 설정에서 학습 시간대/리마인드 옵션(로컬 알림 기반)
- 최근 틀린 단어 강조 노출
- 온보딩(30-60초)으로 제품 사용법 안내

Should 항목은 일정/리스크에 따라 제외될 수 있으며, 제외 시 Must 품질을 우선 강화한다.

---

## 5) Won't Have (MVP 제외)

- 소셜 기능(친구, 랭킹, 공유 피드)
- 과금/구독 시스템(인앱 결제 포함)
- 대규모 콘텐츠 마켓/코스형 커리큘럼
- 웹/데스크톱 동시 정식 출시(우선순위 플랫폼 외)
- 고급 발음 분석(음소 단위 정밀 진단, 원어민 비교 시각화)
- 복잡한 게이미피케이션(배지 다단계, 이벤트 시즌제)

---

## 6) 수용 기준(Definition of Done for MVP)

아래 조건을 모두 만족하면 MVP 범위 달성으로 본다.

1. 신규 사용자가 설치 후 5분 이내 첫 학습을 완료할 수 있다.
2. 사용자는 하루 학습(3-5개)과 당일 복습을 같은 날 수행할 수 있다.
3. 앱 재시작/오프라인 상황에서도 학습 상태와 이력이 유지된다.
4. 스트릭 및 복습 일정이 정책 문서와 일관되게 계산된다.
5. 핵심 플로우에서 치명 오류(진행 불가)가 재현되지 않는다.
6. 핵심 화면 UI는 `17_UI_STYLE_NOTEBOOK_THEME.md`의 톤/컴포넌트 규칙과 일치한다.

---

## 7) 릴리즈 게이트(Go/No-Go)

릴리즈 전 다음 질문에 모두 "예"여야 한다.

- Today -> 학습 -> 복습 큐 생성 -> Inbox 수행까지 단절이 없는가?
- 최소 7일 시뮬레이션에서 복습/스트릭 계산 불일치가 없는가?
- AI 생성 실패 시 사용자가 다음 행동을 취할 수 있는가?
- 크래시/데이터 유실 없이 루틴을 반복할 수 있는가?

하나라도 "아니오"라면 출시를 보류하고 원인 수정 후 재검증한다.

---

## 8) 범위 변경 규칙

- Must 항목 추가는 릴리즈 일정, 품질, 기술 리스크를 함께 갱신할 때만 허용한다.
- Won't 항목을 MVP로 승격할 경우, 기존 Must의 품질 저하가 없는지 먼저 검토한다.
- 범위 변경은 `ssot.md` 변경 절차(문서 -> 테스트 -> 구현)를 따른다.
